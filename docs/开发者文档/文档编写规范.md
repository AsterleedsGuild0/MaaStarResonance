# 文档编写规范

## 目标

- 让用户在 30 秒内判断这篇文档是否适合自己。
- 让维护者新增功能文档时有固定结构可套用。
- 降低 README、入口页、功能页之间的重复描述。

## 通用写法

- 开头先写“这篇文档适合谁”。
- 前 20 行给出结论、限制、前置条件。
- 一个页面只解决一个核心问题。
- 操作步骤使用有序列表。
- 注意事项使用单独的“限制 / 风险 / 常见问题”标题。
- 截图只保留能帮助决策或定位控件的图片。

## 模板一：入口页

1. 这是什么。
2. 我该先看哪一篇。
3. 按场景查找。
4. 注意事项。

## 模板二：功能使用页

1. 功能简介。
2. 使用前提。
3. 配置步骤。
4. 特殊限制。
5. 常见问题。

## 模板三：开发者说明页

1. 适用对象。
2. 前置环境。
3. 操作步骤。
4. 验证方式。
5. 参考资料。

## 术语约定

- “入口页”负责分流和导航，不负责写完整操作细节。
- “功能使用页”只解释一个功能或一组紧密相关的操作。
- “开发者说明页”默认面向已经准备进入仓库修改内容的贡献者。

## 图片使用建议

- 一组连续步骤优先保留 1 到 3 张关键截图。
- 截图文件名尽量与场景对应，避免难以维护的重复图片。
- 如果图片不能帮助用户判断按钮位置、状态差异或配置结果，就不要保留。
